<objective>
配置 Railway 部署所需的基础设施文件，确保容器绑定端口动态化、SQLite 和上传文件持久化、健康检查路径正确。

当前 Dockerfile 的 gunicorn 使用 `"--bind", "0.0.0.0:5001"` 硬编码端口；Railway 会注入 `PORT` env var（通常为 8080），导致 gunicorn 监听 5001 而 Railway 代理打 8080，请求无法到达应用。

Purpose: 消除 Railway 部署的端口绑定 mismatch，建立 volume mount 使 SQLite 和上传图片跨重启持久化，配置健康检查让 Railway 知道服务何时就绪。

Output:
- `Dockerfile` — CMD 改为 shell form，使用 `${PORT:-5001}` 动态绑定
- `railway.json` — 新建 Railway 配置文件（volume + healthcheck + deploy settings）
</objective>

  <action>
将 Dockerfile 末尾的 `CMD` 从 exec form（JSON 数组）改为 shell form（单字符串），使 `${PORT:-5001}` 能被 shell 展开：

**替换前：**
```dockerfile
CMD ["gunicorn", \
     "--chdir", "/app/backend", \
     "--bind", "0.0.0.0:5001", \
     "--workers", "2", \
     "--threads", "4", \
     "--timeout", "300", \
     "--access-logfile", "-", \
     "app:create_app()"]
```

**替换后：**
```dockerfile
CMD gunicorn \
    --chdir /app/backend \
    --bind "0.0.0.0:${PORT:-5001}" \
    --workers 2 \
    --threads 4 \
    --timeout 300 \
    --access-logfile - \
    "app:create_app()"
```

Shell form 由 `/bin/sh -c` 执行，`${PORT:-5001}` 会被展开：PORT 有值时用 PORT，否则 fallback 5001。

不修改 Dockerfile 的其他任何内容。
