# MiroFishmoody UI Review / 信息架构收口

## 先说结论

这套稿子的方向没有偏，但当前还不能直接交前端实现。主要不是“美观不够”，而是语义、层级、token 和组件边界都不稳定。

## 核心问题

### 1. Token 命名不稳定，工程上一定会漂
- 颜色名、类名、语义名混用，后续一接 React 就会出现 `bg-cream`、`cream-bg`、`text-coffee`、`deep-coffee` 并存。
- 本次前端统一只保留一套 token：`cream`、`paper`、`stone`、`line`、`coffee`、`ink`、`mist`、`mist-soft`、`wine`。

### 2. 首页和总览台本质重复，需要合并成一个入口
- 原稿里“首页 / 总览台”是两个名字，一个页面职责。
- 这里直接收口为一个页面：`总览台`。
- 这个页面只做 3 件事：看当前工作队列、看待处理事项、看最近结论，不再混杂品牌介绍和泛平台叙事。

### 3. 语言仍然偏“交易市场”与“技术系统”
- `Probability Board`、`Judge`、`参与人数`、`计票`、`活跃交易头寸` 这类说法都不适合 Moody 内部工作台。
- 统一改为：`相对胜出机会`、`评审摘要`、`当前阶段`、`待结算`、`待校准`、`建议优先测试`、`进入下一轮优化`。
- 运行中页只展示团队关心的真状态：当前阶段、整体进度、当前方案、预计剩余时间。

### 4. 示例数据长得像真数据，风险很高
- 如果没接 API，就不该把具体百分比、精度、CTR/CVR 写得像真实结果。
- 本次实现统一用两种表达：
  - `示例数据`
  - `待实际投放回填`
- 结果页改用定性等级和文案摘要，不再伪造高精度数字。

### 5. 组件边界不清，后续很容易一页一个样
- 原稿更像 5 张独立静态图，不像一个系统。
- 这次实现统一成一套共享组件：
  - 页面壳 `AppShell`
  - 容器卡片 `SectionCard`
  - 状态标记 `StatusBadge`
  - 通用字段输入 `field-*`
- 页面之间共享同一套页头、按钮、标签、卡片节奏。

## 收口后的信息架构

### 1. 总览台
- 看当前有哪些评审在推进
- 看哪些任务待结算 / 待校准
- 看最近结论和本周重点

### 2. 新建评审页
- 填写评审主题与业务背景
- 录入多个 campaign 方案
- 检查提交前清单

### 3. 运行中状态页
- 看当前阶段
- 看阶段进度
- 看当前正在比较的方案与摘要

### 4. 结果页
- 看推荐方案
- 看方案排序和维度差异
- 看受众反馈、两两对比摘要、建议动作

### 5. 结算 / 历史页
- 回填真实结果
- 管理待结算队列
- 查看已完成历史与校准状态

## 这次前端实现的约束

- 中文优先，不再让英文抢主导
- 奶油白 / 深咖 / 雾蓝灰为主，酒红只做提醒
- 不做交易终端感，不做普通 SaaS 后台感
- 所有页面默认可独立阅读，不依赖后端真值也不会误导成真实数据
